# Bee v3 learnings — flat-top hex geometry and corpus-vs-algorithm bottleneck

## Context

v3 shipped the Spelling Bee variant: hand-curated names corpus + active-roster auto-merge, deterministic puzzle generator, hex-grid UI, three routes (`/nba/bee`, `/wnba/bee`, `/all/bee`), GOAT modal, share text. Two non-obvious things came out of the build that future-me will want documented before they fade:

1. The hex grid took two iterations to look right. The first version used pointy-top math and visibly didn't match the NYT reference.
2. Expanding the WNBA corpus by ~35 retired greats did **not** lift the number of valid names per puzzle. The bottleneck was somewhere else.

## Guidance

### Hex geometry: NYT Bee uses flat-top hexes, not pointy-top

A pointy-top hexagon has vertices at the top and bottom. A flat-top hexagon has flat edges at the top and bottom — this is what NYT uses, and it's what makes the central + 6 neighbor layout look symmetric in a square viewport.

The two layouts have different math:

```typescript
// Flat-top (what NYT uses, what we shipped):
const R = 50;                                // hex radius (center to vertex)
const GAP = 3;                               // visual gap between cells
const neighborDistance = R * Math.sqrt(3) + GAP;
const neighborAngles = [90, 30, -30, -90, -150, 150];   // degrees, top → clockwise
const vertexAngles   = [0, 60, 120, 180, 240, 300];     // for SVG polygon points

// Pointy-top (what we tried first, looks wrong):
// neighborAngles = [60, 0, -60, -120, 180, 120]
// vertexAngles   = [30, 90, 150, 210, 270, 330]
```

The key tell that you've picked the wrong orientation: neighbors don't tessellate with the center cell — they sit slightly off-axis and the cluster looks tilted.

### Corpus depth ≠ generator output

Initial intuition: WNBA Bees have only 3–4 valid names because the corpus is too thin. So we expanded it from ~50 → ~150 names (active roster + ~35 hand-curated all-time greats). Result: still 3–4 valid names per puzzle.

Why: the generator picks pangram seeds, then for each pangram enumerates the candidate boards (which 7 letters), then **greedily picks the first board that produces ≥ TARGET_MIN valid names**. With TARGET_MIN=10 and a sparse name distribution, it walks past richer boards to take the first one that clears the floor.

The bottleneck is the search strategy, not the input. Two fixes that would actually move the number:

- **Score boards instead of first-match.** Enumerate N boards per pangram, pick the one with the most valid names (or balanced pangram count + valid count).
- **Lift the floor.** Raise TARGET_MIN, accept lower acceptance rate, retry across more pangrams.

We deferred this to U7 since the v3 product is shippable as-is.

## Why This Matters

**Hex geometry:** if you start with the wrong orientation, every other measurement (gap, viewbox padding, click target alignment) is off by a few pixels and you'll fight it forever. Pick orientation first, lock vertex + neighbor angle pairs, then tune sizing.

**Corpus vs algorithm:** "the data is too thin" is the most natural diagnosis for a constraint-satisfaction generator that produces small outputs, and it's often wrong. Adding data is expensive (curation time, fame-floor decisions, bias risk). Profile the generator first — log how many candidate boards it considered before accepting one. If the answer is "1," you have an algorithm problem, not a data problem.

## When to Apply

- Building a hex grid that needs to mirror an existing reference (NYT, Honeycomb, etc.) — verify orientation against the reference before writing math.
- Tuning a generator that picks the first acceptable output — instrument the search before expanding inputs.
- Triaging a "small output" complaint on any constraint solver — corpus expansion is rarely the cheapest fix.

## Examples

**Hex orientation diagnostic:** render the central cell + just one neighbor at angle 90°. If the shared edge is horizontal, you have flat-top hexes. If it's a single point, you have pointy-top.

**Generator instrumentation:** in [pipeline/nba_mini/bee/generate.py](../../pipeline/nba_mini/bee/generate.py), the loop that picks a board could log `(seed, candidate_count, valid_name_count, accepted)` for every attempt. The first WNBA puzzle would tell you immediately whether the generator considered 1 board or 50.
